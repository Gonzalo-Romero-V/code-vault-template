Qué es

El problema que resuelve

Cuando trabajás con agentes de IA, cada sesión arranca sin contexto: enums inventados, endpoints duplicados, decisiones documentadas que se ignoran. Code Vault fuerza al agente a leer un grafo semántico curado antes de tocar código.

Usalo si...

  • Trabajás con agentes de IA y querés contexto consistente entre sesiones
  • Tu proyecto tiene reglas de negocio que no caben en el código (invariantes, RBAC, decisiones arquitectónicas)
  • Te molesta que la IA invente enums o ignore decisiones documentadas

No lo uses si...

  • El proyecto es un script de una sesión, prototipo descartable o POC de menos de un día
  • El dominio es trivial y todo cabe en el README
  • No tenés tolerancia para mantener notas Markdown como segundo cerebro
Arquitectura

Jerarquía H1 → H5

Las capas superiores causan las inferiores. H1–H3 viven en el vault (texto, decisiones), H4–H5 viven en el código. Cuando una capa superior cambia, las inferiores deben propagar; lo opuesto es un bug.

H1

Intent · El porqué

Visión, invariantes de negocio, reglas que no negocian. Humano-mandado, inmutable salvo decisión deliberada. vault/intent/

H2

Requirements · El qué

Entidades de dominio, estados, ciclos de vida, reglas operativas. vault/domain/ y vault/raw/

H3

Architecture · El cómo

Decisiones de stack, patrones, convenciones que propagan hacia abajo. vault/decisions/

— vault termina aquí · el código empieza —
H4

Contracts · Materialización formal

Modelos, schemas, migraciones, rutas, tipos. Derivado de H1–H3. Refleja, no decide.

H5

Implementation · El código vivo

Controllers, handlers, services, componentes. Nunca contradice H1–H3 silenciosamente.

El ciclo de sincronización

Codeás un cambio

Editás archivos del repo, normalmente en H4 (contratos) o H5 (implementación).

Commit aprobado

Pedís al agente que haga commit; vos aprobás el mensaje. git commit.

Hook post-commit genera artefactos

El hook corre vault_sync.py report, que produce .vault-sync/change_report.json (qué cambió) y .vault-sync/facts.json (qué hay) deterministicamente, sin LLM.

Skill sync propone cambios al vault

El agente lee los dos artefactos y produce .vault-sync/proposed-changes.json con operaciones discretas (update_frontmatter, append_section, create, deprecate).

Aprobás (o ajustás) la propuesta

El humano siempre revisa antes de tocar el vault. Las skills no aplican unilateralmente.

Engine aplica idempotentemente

vault_sync.py apply ejecuta las operaciones. Idempotente, append-only, locked-aware. El vault refleja el código.

Regla de oro: el código (H4–H5) refleja al vault (H1–H3), nunca al revés. Si el código contradice una nota stable o locked, el agente reporta — no resuelve unilateralmente.

Inicio rápido

4 comandos y arrancás

El bootstrapper instala todo: vault Obsidian inicializado, scripts del engine, git hook activo, AGENTS.md generado, vault_sync.config.json con paths reales.

terminal · bash 
# 1. Cloná el template
git clone https://github.com/Gonzalo-Romero-V/code-vault-template.git mi-proyecto
cd mi-proyecto

# 2. (Recomendado) reseteá la historia para empezar limpio
rm -rf .git && git init

# 3. Bootstrappeá con tu stack + nombre + path al vault
python bootstrap/init_vault.py \
  --stack python-fastapi \
  --project-name "MiProyecto" \
  --vault-path "$HOME/vaults/mi-proyecto"

# 4. Primer commit — el hook genera el primer reporte automáticamente
git add . && git commit -m "chore: init code-vault"

Próximos pasos tras el bootstrap

Editá intent/vision.md

Nota H1 — obligatoria antes del primer feature. Visión, invariantes, fuera de alcance.

Editá decisions/stack.md

Nota H3 — qué tecnologías y por qué. Sin alternativas descartadas, la decisión es frágil.

Editá decisions/architecture.md

Patrón arquitectónico, separación de responsabilidades, naming, manejo de errores.

Completá la tabla "tarea → notas obligatorias" en INDEX.md

Esta tabla es la mitad del valor del sistema. Sin ella, los agentes no saben qué leer antes de tocar código.

Flujo diario

Una sesión típica

El uso cotidiano del sistema cabe en 6 pasos. Las skills son propositivas: nunca aplican sin tu aprobación explícita.

Recibís una tarea

El agente lee AGENTS.md y la tabla "tarea → notas obligatorias" para identificar qué notas del vault debe leer antes de tocar código.

Implementás el cambio

El agente codea respetando las invariantes documentadas. Si encuentra una contradicción, reporta — no resuelve.

Commit aprobado

Pedís al agente confirmación, aprobás mensaje, git commit. El hook genera change_report.json y facts.json.

Corrés la skill sync

El agente lee los artefactos, identifica qué notas actualizar, te muestra la propuesta. Si el commit fue refactor puro H5, la propuesta puede ser 0 cambios — eso es válido.

Aprobás o ajustás

Revisás .vault-sync/proposed-changes.json. Aprobás o pedís cambios.

Engine aplica idempotentemente

El vault refleja el código. Si volvés a aplicar el mismo changes.json, no se duplica nada.

Estados de las notas: draft (libre), stable (solo append + code_path), locked (intocable, solo edición humana), deprecated (histórico). El engine respeta automáticamente locked y nunca borra notas.

Skills

Las 4 operaciones del sistema

Cada skill tiene una intención clara. En Claude Code se invocan como slash commands (/sync, /check, /ingest, /snapshot). En otros CLIs (Codex, Cursor, Cline) están documentadas de forma agnóstica en prompts/SKILLS.md.

sync — sincronización tras commit

Costo en tokens: medio

Es el flujo natural de mantenimiento del grafo. Después de cada commit aprobado, el agente lee .vault-sync/change_report.json y .vault-sync/facts.json, identifica qué notas del vault deben actualizarse, y produce una propuesta concreta. Vos aprobás, el engine aplica.

Qué hace

  • Valida el reporte contra el schema.
  • Para cada nota en vault_hints.potentially_affected, decide si actualizar code_path o agregar sección "Implementación".
  • Detecta si el commit incorpora capas nuevas que necesitan documentación.
  • Reporta contradicciones detectadas — no las resuelve.
  • Genera .vault-sync/proposed-changes.json con operaciones discretas.

Cuándo invocarla

Después de cada commit que toque dominio o decisiones técnicas. Si fue refactor puro de implementación (H5), sync puede proponer 0 cambios — eso es válido.

check — auditoría de coherencia

Costo en tokens: bajo

Auditoría transversal entre el vault y el repo. Combina chequeos deterministas del engine con interpretación semántica del agente. Solo reporta — nunca resuelve.

Qué chequea

  • env_references: variables usadas en código (process.env.X, os.getenv('Y'), env('Z')) que no aparecen declaradas en .env.example.
  • vault_code_paths: notas con code_path: que apuntan a archivos inexistentes.
  • Coherencia semántica por pares H1↔H2, H2↔H3, H3↔H4, H4↔H5.
  • Frescura de Graphify (si está activo): compara mtime de graphify-out/graph.json con el último commit.

Output

Tres categorías: 🔴 Contradicciones (regla X dice A, código hace B) · 🟡 Drift (concepto sin reflejo en código o viceversa) · 🟢 Coherente.

ingest <ruta> — incorporar documento nuevo

Costo en tokens: medio–alto (depende del doc)

Cuando dropeás un PDF, .txt, imagen o transcript en vault/raw/, esta skill extrae el contenido semántico y propone notas estructuradas en intent/, domain/ o decisions/.

Qué hace

  • Identifica el tipo de información (contexto, decisión, referencia).
  • Extrae entidades, reglas y relaciones.
  • Detecta si hay contradicciones con reglas existentes — las reporta, no las resuelve.
  • Propone create para conceptos nuevos o append_section para enriquecer notas existentes.
  • Nunca deprecia automáticamente — esa decisión es del humano.

Cuándo invocarla

Inmediatamente después de agregar un archivo a vault/raw/. El contenido semántico se mueve a notas tipadas; los archivos en raw/ quedan como referencia.

snapshot — mapeo completo inicial

Costo en tokens: alto · usar con criterio

Regenera el mapeo completo concepto ↔ código. Operación cara en tokens, reservada para tres escenarios.

Cuándo invocarla

  • Al iniciar un proyecto nuevo con código pre-existente.
  • Después de un refactor masivo.
  • Cuando sospechás que el grafo perdió sincronía con el código.

Qué hace

  • Si graphify-out/graph.json existe, lo usa como mapa estructural.
  • Lee facts.json para conocer el estado semántico actual.
  • Mapea cada entidad H4 detectada al vault, proponiendo crear o actualizar notas.
  • Identifica gaps: vars de entorno sin documentar, decisiones implícitas, entidades sin nota domain.
Engine

Comandos manuales (sin LLM)

El engine determinista es invocable directamente. Útil para CI, debugging, o cuando estás operando sin agente.

Comando Qué hace Exit codes
vault_sync.py report Genera change_report.json + regenera facts.json con el último commit. 0 OK · 2 excede max_files · 3 schema inválido
vault_sync.py facts Regenera solo facts.json con el extractor activo. 0 OK
vault_sync.py validate <file> Valida un change_report.json contra el schema fijo. 0 OK · 1 file not found · 3 schema inválido
vault_sync.py check Chequeos deterministas: env_references + vault_code_paths. 0 OK · 4 consistency failures
vault_sync.py apply <changes.json> Aplica un changes.json al vault. Idempotente, append-only, respeta locked. 0 OK · 5 ops fallidas
vault_sync.py status Imprime estado: extractor activo, notas, último report, último facts. 0 OK

Sistema de extractores

El campo extractor en vault_sync.config.json decide qué plugin produce facts.json. Cambiar el extractor no cambia el schema del report ni la semántica del vault — solo cambia qué metadata se captura.

nextjs-laravel

Implementación de referencia. Extrae componentes shadcn, tsconfig paths, tema Tailwind, package.json, modelos Eloquent (table + fillable), migrations, rutas, controllers, services.

python-generic

Cubre FastAPI (decoradores), Pydantic (BaseModel), SQLAlchemy (__tablename__, declarative_base), Alembic (versions), Flask (@app.route), Django (manage.py + apps).

graphify

Adaptador para Graphify: lee graphify-out/graph.json y expone call_graph (función-a-función, exclusivo) en addition al import_graph. 25 lenguajes vía tree-sitter.

generic

Fallback inspector + scaffolder. Detecta el stack por signal files (angular.json, pom.xml, Cargo.toml, go.mod, Gemfile, etc.) y genera scaffolds _proposed_* para promoverlos al catálogo.

FAQ

Preguntas frecuentes

¿Es agnóstico al CLI de IA que use?

El engine es 100% agnóstico. Ningún archivo bajo scripts/, bootstrap/, hooks/ o tests/ menciona Claude Code. El protocolo AGENTS.md es la convención multi-CLI emergente (lo lee Codex CLI nativamente).

Las skills como slash commands (/sync, etc.) son específicas de Claude Code. Para Codex / Cursor / Cline / Continue, las mismas 4 operaciones están documentadas en prompts/SKILLS.md en formato neutro.

¿Necesito Obsidian para usarlo?

No estrictamente. El vault es una carpeta con archivos .md que respetan frontmatter YAML. Cualquier editor sirve para verlos. Pero Obsidian (o cualquier app con soporte de wikilinks [[nota]] y grafo) maximiza el valor del segundo cerebro.

¿Por qué el vault no está dentro del repo?

Por separación de propiedad. El repo lo versionás con tu equipo en GitHub; el vault es tu segundo cerebro personal que tal vez sincronizás con Obsidian Sync o iCloud, comparte con stakeholders no técnicos, o crece más allá del scope del proyecto técnico. Mantenerlos separados evita commits ruidosos y permite políticas de acceso distintas.

¿Qué pasa si la IA propone modificar una nota locked?

El engine bloquea automáticamente esa operación con código BLOCKED (locked) en el log. La nota no se toca. El protocolo manda que el agente reporte la situación al humano: solo edición manual puede modificar notas locked (típicamente SYSTEM.md e INDEX.md).

El hook bash no se ejecuta en Windows. ¿Qué hago?

El hook necesita un intérprete bash. En Git for Windows (default desde el instalador oficial), msys2 lo interpreta sin problemas. Si usás Git via Scoop sin msys o tenés otra instalación sin bash en PATH, corré manualmente python scripts/vault_sync.py report tras cada commit, o configurá tu Git para usar el bash de Git for Windows.

¿Cómo agrego un stack nuevo (Java, Go, Rust, Angular)?

Bootstrappeá con --stack generic. El extractor generic detecta el stack por signal files y, si no está en el catálogo, escribe dos scaffolds:

  • bootstrap/stacks/_proposed_<stack>.json — hierarchy_mapping inferido.
  • scripts/lib/extractors/_proposed_<stack>.py — clase Extractor con TODOs marcados.

Editás los parsers específicos, renombrás quitando _proposed_, registrás la clase en scripts/lib/extractors/__init__.py::REGISTRY. El catálogo crece cuando vos usás la pila en serio, no antes.

¿Cuánto tokens consume cada skill?
  • /check — Bajo. Mayormente determinista.
  • /sync — Medio. Lee 2 artefactos + notas hint, propone cambios.
  • /ingest — Medio–Alto. Depende del documento (un PDF grande puede ser caro).
  • /snapshot — Alto. Lee facts.json + vault completo. Usar solo en arranque o refactor masivo.

El grueso del trabajo lo hace el engine determinista (0 tokens). El LLM entra solo donde hace falta razonar sobre semántica.

¿Qué hace exactamente el post-commit hook?

Después de cada commit, el hook detecta el binario de Python disponible (python, python3 o py en Windows), corre scripts/vault_sync.py report y registra el output en .vault-sync/post-commit.log. Es non-blocking: nunca aborta un commit, siempre sale con código 0. Si algo falla, queda registrado en el log para diagnóstico.

¿Cómo corro los tests del sistema?

Suite stdlib, sin dependencias externas:

python -m unittest discover -s tests -t .

Cubre hierarchy (glob matching), vault (frontmatter parser + update preservando YAML), apply (las 4 actions + locked guard + idempotencia), schema (validación del report). 43 tests, todos OK.

¿Puedo cambiar de extractor sin re-bootstrappear?

Sí. Editá vault_sync.config.json y cambiá el campo extractor. Después corré python scripts/vault_sync.py facts para regenerar facts.json con el nuevo extractor. El change_report.json no cambia — su schema es universal.
